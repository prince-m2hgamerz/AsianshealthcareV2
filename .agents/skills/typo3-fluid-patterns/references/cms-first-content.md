# CMS-First Content Architecture

## lib.dynamicContent — The Foundation

Every piece of editor-managed content should be rendered via `lib.dynamicContent`. This TypoScript object queries `tt_content` by column position:

```typoscript
lib.dynamicContent = CONTENT
lib.dynamicContent {
  table = tt_content
  select {
    orderBy = sorting
    where = {#colPos}={field:colPos}
    where.insertData = 1
  }
  slide.field = slide
}
```

### Usage in Fluid

```html
<!-- Render main content area (colPos 0) -->
<f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 0}"/>

<!-- Render sidebar (colPos 1) -->
<f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 1}"/>

<!-- Render footer column with content sliding (inherited from parent pages) -->
<f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 11, slide: '-1'}"/>
```

### The `slide` Parameter

`slide: '-1'` enables **content sliding** — if no content is defined for this `colPos` on the current page, TYPO3 traverses up the page tree until it finds content. Used for footer columns so editors define footer content once on the root page.

## colPos Conventions

| colPos | Purpose | Example |
|--------|---------|---------|
| 0 | Main content area | Primary page content |
| 1 | Sidebar | Secondary content in sidebar layout |
| 2 | Stage / Hero | Full-width hero area above main content |
| 10 | Footer bar | Copyright, legal links (bottom row) |
| 11–14 | Footer columns 1–4 | Logo, menus, contact info |

> Ranges are project-specific. Define your conventions early and document them.

## lib.contentElement — Content Element Rendering

This defines how individual content elements are rendered:

```typoscript
lib.contentElement = FLUIDTEMPLATE
lib.contentElement {
  layoutRootPaths.20 = EXT:theme_name/Resources/Private/Templates/Layouts/
  partialRootPaths.20 = EXT:theme_name/Resources/Private/Templates/Partials/
  templateRootPaths.20 = EXT:theme_name/Resources/Private/Templates/Content/

  # Auto-resolve template name from CType (e.g., "text" → "Text.fluid.html")
  templateName >
  templateName.ifEmpty.cObject = TEXT
  templateName.ifEmpty.cObject {
    field = CType
    required = 1
    case = uppercamelcase
  }

  dataProcessing.1421884800 = record-transformation
}

tt_content = CASE
tt_content.default =< lib.contentElement
```

**Key concept:** The `templateName` is auto-resolved from the CType field using `uppercamelcase`. So `CType = mysitepackage_hero` becomes `MysitepackageHero.fluid.html`.

## Content Layout: Frame Classes

The content layout (`Layouts/Content/Default.fluid.html`) handles background variants (frame classes) set by editors in the backend:

```html
<f:switch expression="{data.frame_class}">
    <f:case value="bg-10"><f:variable name="frame" value="bg-10"/></f:case>
    <f:case value="bg-80"><f:variable name="frame" value="bg-80"/></f:case>
    <f:case value="bg-100"><f:variable name="frame" value="bg-100"/></f:case>
</f:switch>

<f:render partial="Wrapper/Component" arguments="{frame: frame}" contentAs="content">
    <f:render section="Header" optional="true">
        <f:render partial="Header/Header" arguments="{
            header: data.header,
            layout: data.tx_mysitepackage_header_style,
            tag: data.header_layout,
        }"/>
    </f:render>
    <f:render section="Main" optional="true"/>
</f:render>
```

The `Wrapper/Component` partial then applies the frame class to the wrapping element, enabling editors to choose background variants for any content element.

## Content Layout Variants

| Layout | When to Use |
|--------|-------------|
| `Content/Default` | Standard content elements with frame, header, and main sections |
| `Content/Unwrapped` | Full-bleed elements like Hero that manage their own wrapping |

Content templates select their layout:

```html
<!-- Standard content element -->
<f:layout name="Content/Default"/>

<!-- Full-bleed element (Hero) -->
<f:layout name="Content/Unwrapped"/>
```

## RTE Customization

Customize how the Rich Text Editor output is rendered via `lib.parseFunc_RTE`:

```typoscript
lib.parseFunc_RTE {
  externalBlocks := addToList(ol,ul)
  externalBlocks {
    ol {
      stripNL = 1
      callRecursive = 1
      callRecursive.tagStdWrap {
        HTMLparser = 1
        HTMLparser.tags.ol.fixAttrib.class.default = list list--ordered
      }
    }
    ul {
      stripNL = 1
      callRecursive = 1
      callRecursive.tagStdWrap {
        HTMLparser = 1
        HTMLparser.tags.ul.fixAttrib.class.default = list list--unordered
      }
    }
  }
}
```

This automatically adds CSS classes to `<ol>` and `<ul>` elements generated by the RTE.

## Footer: The CMS-First Way

**Anti-pattern:** Implementing footer content via TypoScript settings/variables. This defeats the purpose of a CMS — editors can't change the footer without developer intervention.

**Correct approach:** Use `lib.dynamicContent` with dedicated footer `colPos` values and `slide: '-1'` for inheritance:

```html
<footer>
    <div class="grid grid-cols-1 md:grid-cols-4 gap-l">
        <!-- Column 1: Logo & About -->
        <div class="footer__col">
            <f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 11, slide: '-1'}"/>
        </div>
        <!-- Column 2: Menu 1 -->
        <div class="footer__col">
            <f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 12, slide: '-1'}"/>
        </div>
        <!-- Column 3: Menu 2 -->
        <div class="footer__col">
            <f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 13, slide: '-1'}"/>
        </div>
        <!-- Column 4: Contact -->
        <div class="footer__col">
            <f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 14, slide: '-1'}"/>
        </div>
    </div>

    <!-- Footer bar: copyright / legal -->
    <div class="border-t border-white/10">
        <f:cObject typoscriptObjectPath="lib.dynamicContent" data="{colPos: 10, slide: '-1'}"/>
    </div>
</footer>
```

This way, editors manage all footer content through the TYPO3 backend, and the `slide` parameter ensures child pages inherit the root page's footer content.
